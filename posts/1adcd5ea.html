redisson生产场景（粉丝投稿）

锁 redisson

发布日期:   2023-07-24
更新日期:   2023-11-13

场景一：

近期项目上面有个功能出现bug. 批量登记模型，这个模型名称有必须唯一性 在批量登记都会校验模型名称不能重复，测试的同志在测试不知道不是系统卡还是怎么了，成功添加了同名的模型名称但是不同批次号。但是在流程办结的时候流程的状态更新到数据库其还是会调用那个批量登记方法，这个方法就有校验模型唯一性，导致这个比业务不能顺利走完流程。

由于项目是集群的所以。就这里已经排除jvm层面的上锁情况。考虑的是分布式锁的。笔者还是通过多方面的选择最

第一要批量生成多个对象锁，第二同一批次的锁也不能同名。那么分布式锁使用的结构正hash数据结构:

终选择redisson框架的锁。来完成本项目需求的缺陷。

1、两种引入方式

方式一推荐
<!-- https://mvnrepository.com/artifact/org.redisson/redisson-spring-boot-starter -->
        <dependency>
            <groupId>org.redisson</groupId>
            <artifactId>redisson-spring-boot-starter</artifactId>
            <version>3.17.1</version>
        </dependency>
方式二  <!-- 原生-->
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson</artifactId>
    <version>3.17.1</version>
</dependency>

3、引入redisson依赖

只需引入redisson-spring-boot-starter就可以了，不过这里需要注意springboot与redisson的版本，因为官方推荐redisson版本与springboot版本配合使用。版本不匹配会导致这种问题。

将 Redisson 与 Spring Boot 库集成。取决于Spring Data Redis模块，支持 Spring Boot 1.3.x - 2.4.x

这句话是官方说的，不过现在的2.5.x也是支持的，只需要注意springboot最低版本不要低于1.3.x即可。

以上配置参考网络的。但笔者还在使用config配置类的方式实现配置。这样不用维护那么多配置文件。

//单机模式
@Configuration
public class RedissionConfig {
    @Value("${spring.redis.host}")
    private String redisHost;

    @Value("${spring.redis.password}")
    private String password;

    private int port = 6379;

    @Bean
    public RedissonClient getRedisson() {
        Config config = new Config();
        //这里单机模式，后期可以指定集群模式配置
        config.useSingleServer().
                setAddress("redis://" + redisHost + ":" + port).
                setPassword(password);
        config.setCodec(new JsonJacksonCodec());
        return Redisson.create(config);
    }
}

//集群模式
@Configuration
public class RedissionConfig {
    Config config = new Config();
	config.useClusterServers()
    .setScanInterval(2000) // cluster state scan interval in milliseconds
    .addNodeAddress("127.0.0.1:7000", "127.0.0.1:7001")
    .addNodeAddress("127.0.0.1:7002");
        config.setCodec(new JsonJacksonCodec());
        return Redisson.create(config);
    }
}

5、锁失效问题

分布式锁的和@Transactional 失效问题

由于分布式在事务内，当事务未提交但是锁被释放了，事务未提交。其他线程进来来了读取到脏数据。

处理方案可以在

// 伪代码
@Transactional
   public void update(int id) {
       boolean lock = redisLock.lock(id);
       if (!lock) {
           throw new RuntimeException("当前人数过多，请稍后再试");
       }
       /*
         业务代码在该区域
        */
       redisLock.unlock(id);
   }

我们可以在update方法之前就加上锁，在还没有开事务之前就加锁，那么就可以保证线程的安全性，从而不会出现脏读和数据不一致性等情况.

//伪代码  
@RequestMapping("/execute")
    public void execute(int id) {
        boolean lock = redisLock.lock(id);
        if (!lock) {
            throw new RuntimeException("当前人数过多，请稍后再试");
        }
        service.update(id);
        redisLock.unlock(id);
    }
//伪代码
@Transactional
    public void update(int id) {
        /*
          业务代码在该区域
         */
    }

			log.info("{}：执行结束", threadName);
			return ResultUtils.success();
		} catch (Exception e) {
			log.error("testMultLock exception:", e);
			return ResultUtils.sysError();
		} finally {
			// 注意：不能使用isLocked()和isHeldByCurrentThread()方法，会抛出UnsupportedOperationException异常
			multiLock.unlock();
			log.info("{}：释放锁成功", threadName);
		}
	}
